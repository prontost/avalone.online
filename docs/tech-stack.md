# Tech stack — предварительный выбор для Avalone

> Это не финальное решение, а отправная точка для обсуждения.
> Каждый пункт — с trade-offs, чтобы автор идеи мог принять осознанное решение.

## Критерии выбора

1. **Скорость разработки MVP.** Нужно быстро проверить первый сценарий (поездки на работу).
2. **Масштабируемость.** Архитектура должна расти от развозки до метавселенной без переписывания ядра.
3. **Графовые возможности.** Люди, организации, места, возможности — всё связано.
4. **Real-time.** Статусы машин, смен, мест — должны обновляться быстро.
5. **Мобильность.** Пользователи будут в основном со смартфонов.
6. **Инфраструктура оператора.** MacBook + Cloudflare tunnel уже работает для Counta. Можно переиспользовать.
7. **Стоимость.** На старте — минимальные затраты, open-source first.

## Frontend

### Вариант A: PWA на SvelteKit
- **Плюсы:** минимальный bundle size, быстрая загрузка, отличная производительность, встроенная поддержка PWA, простой синтаксис.
- **Минусы:** меньшая экосистема, чем у React; меньше разработчиков на рынке.
- **Когда выбирать:** если приоритет — скорость и производительность на мобильных устройствах.

### Вариант B: PWA на React / Next.js
- **Плюсы:** огромная экосистема, много библиотек компонентов, легко найти разработчиков, Next.js даёт SSR/SSG.
- **Минусы:** бóльший размер bundle, сложнее оптимизация под мобильные.
- **Когда выбирать:** если планируется сложный интерфейс с множеством интеграций.

### Вариант C: PWA на Vue / Nuxt
- **Плюсы:** гибкость, хороший баланс между простотой и мощностью.
- **Минусы:** Меньше готовых библиотек по сравнению с React.
- **Когда выбирать:** если команда знакома с Vue.

### Рекомендация для MVP
**SvelteKit** — из-за фокуса на производительность, малый размер и скорость разработки. Если понадобится перейти на React — логика API останется той же.

## Backend

### Вариант A: Python FastAPI
- **Плюсы:** уже используется в Counta, асинхронный, быстрый, автоматическая документация API, отличная интеграция с ML/AI.
- **Минусы:** меньше готовых решений для real-time, чем у Node.js.
- **Когда выбирать:** если оператор и команда знакомы с Python.

### Вариант B: Node.js / Express или NestJS
- **Плюсы:** идеален для real-time (WebSockets), один язык с frontend.
- **Минусы:** для AI/ML придётся подключать Python отдельно.
- **Когда выбирать:** если приоритет — real-time и full-stack JavaScript.

### Рекомендация для MVP
**Python FastAPI** — потому что уже есть инфраструктура и опыт с Counta. Real-time решается через WebSockets/SSE.

## База данных

### Реляционная: PostgreSQL
- **Плюсы:** надёжность, сложные запросы, JSONB для гибких данных, хорошо знакома по Counta.
- **Минусы:** для глубоких графовых запросов может быть неоптимальна.

### Графовая: Neo4j
- **Плюсы:** идеальна для связей Человек — Организация — Возможность — Репутация.
- **Минусы:** отдельная база, отдельный язык запросов (Cypher), дополнительная инфраструктура.

### Гибрид: PostgreSQL + pg_graphql / Apache AGE
- **Плюсы:** одна база, графовые возможности поверх PostgreSQL.
- **Минусы:** менее зрелые, чем Neo4j.

### Рекомендация для MVP
**PostgreSQL** на старте. Графовые связи можно моделировать через таблицы связей (adjacency list / closure table). При росте — добавить Neo4j или Apache AGE.

## Real-time

### Вариант A: WebSockets
- **Плюсы:** двусторонняя связь, низкая задержка.
- **Минусы:** сложнее масштабировать при большом числе пользователей.

### Вариант B: Server-Sent Events (SSE)
- **Плюсы:** проще, если нужно только push от сервера к клиенту.
- **Минусы:** нет обратного канала.

### Вариант C: Firebase / Supabase Realtime
- **Плюсы:** готовое решение, быстрый старт.
- **Минусы:** vendor lock-in, зависимость от внешнего сервиса.

### Рекомендация для MVP
**WebSockets через FastAPI** для начала. Если нагрузка вырастет — перейти на Redis + WebSockets или специализированный real-time сервис.

## Карты и гео

### Вариант A: Kakao Maps API
- **Плюсы:** лучшее покрытие Кореи, привычна местным пользователям.
- **Минусы:** требует регистрации приложения, документация на корейском.

### Вариант B: Naver Maps API
- **Плюсы:** точные адреса в Корее.
- **Минусы:** аналогично Kakao.

### Вариант C: OpenStreetMap + Leaflet
- **Плюсы:** бесплатно, open-source, нет привязки к вендору.
- **Минусы:** менее точные адреса в Корее, нужно самому поддерживать.

### Рекомендация для MVP
**Kakao Maps API** — потому что пользователи в Корее и точность критична. В будущем можно добавить fallback на OSM.

## Auth / Identity

### Вариант A: Phone verification (SMS)
- **Плюсы:** просто, понятно, подходит для мигрантов.
- **Минусы:** стоимость SMS, риск фейковых номеров.

### Вариант B: Kakao / Naver / Google OAuth
- **Плюсы:** быстрый вход, привычно для Кореи.
- **Минусы:** не все мигранты имеют корейские аккаунты.

### Вариант C: Firebase Authentication
- **Плюсы:** готовый phone + OAuth, быстрый старт.
- **Минусы:** vendor lock-in.

### Рекомендация для MVP
**Firebase Authentication** на старте: phone + Kakao/Google. В будущем — миграция на собственную систему с поддержкой DID/VC.

---

## Payments

### Вариант A: Toss Payments
- **Плюсы:** популярна в Корее, удобный API.
- **Минусы:** требует корейской компании.

### Вариант B: Stripe
- **Плюсы:** международная, хороший API.
- **Минусы:** не все способы оплаты доступны в Корее.

### Вариант C: KakaoPay / NaverPay
- **Плюсы:** привычны в Корее.
- **Минусы:** интеграции только для корейских юрлиц.

### Рекомендация для MVP
**Stripe** на старте для простоты. При масштабировании в Корее — добавить Toss Payments / KakaoPay.

## Хостинг / CI/CD

### Текущая инфраструктура оператора
- MacBook + Cloudflare tunnel уже работает для Counta.
- Можно развернуть первую версию Avalone на том же MacBook.
- Потом перейти на VPS (Hetzner, DigitalOcean, AWS Lightsail).

### CI/CD
- GitHub Actions для тестов и сборки.
- Docker для упрощения деплоя.

## Рекомендуемый стек для MVP

| Слой | Технология | Почему |
|------|-----------|--------|
| Frontend | SvelteKit PWA | Быстрота, производительность, мобильность |
| Backend | Python FastAPI | Уже используется в Counta, async, AI-ready |
| Database | PostgreSQL | Надёжность, знакомство, JSONB |
| Real-time | WebSockets (FastAPI) | Низкая задержка для статусов |
| Maps | Kakao Maps API | Лучшее покрытие Кореи |
| Auth | Firebase Auth | Phone + OAuth, быстрый старт |

| Hosting | MacBook + Cloudflare tunnel → VPS | Используем текущую инфраструктуру |

## Следующие технические шаги

3. Подключить Firebase Auth (phone).
4. Сделать прототип карты с точками сбора.
5. Сделать прототип статуса "еду / на месте / уехал".
