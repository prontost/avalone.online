# Инструкции для агентов, работающих с Avalone

## 1. Архитектура: классы + DI в первую очередь

- **Не пиши процедурную бизнес-логику в свободных функциях.** Любое поведение, зависящее от состояния или от других компонентов, должно жить внутри класса.
- **SQL — только внутри `*Repository`.** Не допускай сырого SQL в сервисах, роутах, виджетах и утилитах. Исключение — инфраструктурные миграции в `database.py`/`db.py`.
- **Сервисы получают зависимости через конструктор.** Если сервис нужен роуту — добавь фабрику в `web/api/dependencies.py` (или аналогичный файл пакета) и используй `Depends()`.
- **Не используй глобальные состояния/контекст-переменные как основной способ передачи данных.** Tenant-контекст в finance — допустимое исключение для изоляции между пользователями, но бизнес-сервисы должны работать и с явным `tenant_id`, где это возможно.
- **Если видишь процедурный модуль, который дублирует логику — не дополняй его.** Сначала обсуди вынесение в класс. Если нужна обратная совместимость, оставь модуль как тонкий фасад над сервисом.

## 2. Не хардкоди

- **URL-ы и домены** берутся из `settings().web_base_url` / `settings().api_base_url`. Не хардкодь `https://avalone.online/`.
- **Строки UI** — из глоссария (`glossary.t(...)`). Не вставляй русский/английский текст прямо в шаблоны или JSON-ответы.
- **Лимиты, размеры, флаги** — из `ConstantsService` (finance) или `SettingsRepository` (landing). Не используй магические числа в роутах.
- **Почтовые/ SMTP-настройки** — через `NotifyService`/`MailService` и `SettingsRepository`.

## 3. UI: виджеты, а не inline-HTML

- **Страницы собираются из виджетов.** Базовый класс — `avalone_core.ui.widgets.Widget`.
- **Shell и его части** (`SearchOverlay`, `AppSwitcher`, `ProfileMenu`, `InviteModal`, `FeedbackModal`, `BottomNav`, `NavSidebar`) уже существуют. Используй их, не дублируй HTML.
- **Новые повторяющиеся блоки интерфейса** выноси в отдельный виджет с `template_name` и `context()`.
- **Рендеринг шаблонов** — через `widget.render(env, request)`, а не `templates.TemplateResponse(...)` с захардкоженным контекстом.

## 4. Создавай себе skill-ы для повторяющихся паттернов

- Если ты применяешь один и тот же паттерн больше двух раз (например, «новый сервис + репозиторий + фасад», «новый виджет», «новый FastAPI-роут с DI»), **создай или обнови skill** в `~/.kimi-code/skills/`, а не копируй код.
- Skill должен содержать:
  - краткое описание, когда использовать;
  - минимальный шаблон (код);
  - ссылки на существующие примеры в проекте (файл + строка);
  - список типичных ошибок (циклические импорты, await на sync-метод и т.п.).
- Приоритет: `Project` skill → `User` skill → `Extra` → `Built-in`. Если паттерн специфичен для Avalone, сохраняй его как user-scope skill (`~/.kimi-code/skills/<name>/SKILL.md`) или, если важен для всех агентов в репозитории, добавь в `AGENTS.md`.

## 5. Инструменты и библиотеки, которые уже используются

- **Python 3.13**, управление зависимостями — `uv`.
- **FastAPI** + `python-multipart`. Роуты — `APIRouter`, зависимости — `Depends`.
- **Jinja2** для HTML-шаблонов; шаблоны виджетов — в `templates/widgets/`.
- **SQLite** через `database.py`/`connection()`; миграции — в `database.py` и `db.py`.
- **pytest** — `uv run python -m pytest -q`.
- **Локальный запуск** — `uv run python -m uvicorn avalone_landing.web.app:app --host 127.0.0.1 --port 8000`.
- **Браузер/UI smoke** — `kimi-webbridge` (skill) на `http://127.0.0.1:10086`.

## 6. Что делать перед коммитом

1. `uv run python -m pytest -q` — должен быть зелёным.
2. `uv run python -m uvicorn ...` — приложение должно стартовать без ошибок.
3. Если изменял UI — проведи минимальный smoke через браузер (открыть `/`, `/finance/`, проверить клики/формы).
4. Не коммить `.env`, пароли, тестовые аккаунты.
5. Коммит-сообщение: `refactor(scope): ...`, `feat(scope): ...`, `fix(scope): ...`.

## 8. Важные ограничения

- **Не создавай пользователей с известными паролями в коде/артефактах.** Аккаунты для тестов и админов создаёт сам пользователь.
- **Не удаляй старые фасады**, если от них зависят тесты или другие модули, пока не перепишешь всех потребителей.
- **Не меняй публичные API роутов** без согласования: URL, названия query-параметров и формат ответа остаются прежними.
